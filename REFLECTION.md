## Q1 — Prompting Styles — 在題目一三種風格中，你發現自己最容易卡的是什麼？

> *In the three prompting styles from this assignment, what did you find yourself getting stuck on most often?*

**Style A（One-liner）** 的難點不在於技術，而在於精準：要用一句話同時交代輸入、輸出與意圖，遠比想像中困難。我通常還是會多說一點，才感覺交代清楚。但有趣的是，成果往往比我預期的好——Claude 會自行補上我沒明說的合理細節。

**Style B（Specification）** 是三者中對「下指令的人」要求最高的。你必須清楚知道有哪些處理步驟、哪些資料雜質需要排除、結果應該如何呈現。因為這種風格非常忠實地回應規格，能力強的人能精準控制每個細節，但若對流程不熟，就很難給出完整的 spec。

**Style C（Plan-first）** 反而是最驚喜的。Claude 想到的往往比我周到，會主動提出我沒想到的邊界條件與結構。不過也正因為如此，需要設計幾個確認點——在確認方向正確之後再執行，以免成果偏離原意。對於還不熟悉完整流程的階段，這種風格是很好的起點。

## Q2 — README reflection — 寫 README 時，哪個區塊最難寫？為什麼？

> *When writing the README, which section was hardest and why?*

最難的是 **Title + Tagline** 與 **Motivation**。這兩個區塊沒有現成的結構可以依循，需要自己想清楚「這個專案在說什麼」、「我為什麼做這件事」——而這正是最需要主觀判斷的地方，Claude 沒辦法替我決定。

相比之下，How to Run、Project Structure 這些區塊在有 Claude 協助之後變得非常清晰有條理。我發現，如果一開始的資料夾架構（scaffold）做得好，Claude 會自動追蹤每個步驟、協助補全、甚至提醒遺漏——整個結構會很自然地浮現出來。

不過這也讓我意識到一件事：我並不是對每個步驟都真的很熟悉。過程中我一直搭配著課程簡報對照，才能確認自己沒有做錯。這提醒我，「能用 Claude 完成任務」和「真的理解自己在做什麼」之間，仍然有距離需要填補。

## Q3 — Public artifact reflection — 想像三個月後你要找實習，你會把這個 repo URL 放進履歷嗎？如果不會，缺什麼？

> *Imagine you're applying for internships three months from now — would you include this repo URL? If not, what's missing?*

**會放，但需要再準備。**

這個 repo 的結構完整、有文件、有視覺化成果，作為「可重現分析」的展示作品已經有一定的說服力。但如果真的要在面試場合被問到，我希望自己能流暢地解釋每一個決策——為什麼選這份資料、為什麼這樣清理、圖表背後的結論代表什麼意義。

目前的狀態是：頁面很吸引人，但我對流程的掌握度還需要更鞏固。我希望能再多練習幾遍這一整套步驟，讓它內化成真正屬於自己的能力，而不是依賴 AI 才能完成。

此外，若要讓這個 repo 更有個人辨識度，我想加入更多種類的分析（例如跨題目的知識領域比較、答題時間的探索），以及更能反映我自己研究興趣的詮釋角度——這樣才能真正和造訪 repo 的人產生有意義的對話。
